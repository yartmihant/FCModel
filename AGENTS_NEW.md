# AGENTS.md — агентский протокол проекта FCModel

Этот файл задаёт правила для ИИ-агентов, работающих с проектом **fc_model**.

Проект использует протокол состояния и истории:

- **Состояние** описывает, чем проект является сейчас: структуру, публичный API, источники истины, архитектурные инварианты, правила кодирования, тестов и документации.
- **История** описывает, как проект допустимо менять: через планы в `work/`, изменения в рабочем дереве, обновление тестов и `CHANGELOG.md`, проверки и ручное закрытие пользователем через git-коммит.

Главная цель файла — позволить новой LLM-сессии быстро понять проект, подтянуть нужный контекст и действовать без разрушения API, формата данных, истории и пользовательского контроля.

# I. Состояние проекта

## 1. Сущность проекта

**fc_model** — строго типизированная Python-библиотека объектной модели формата Fidesys Case (`.fc`).

Формат `.fc`:

- JSON;
- кодировка UTF-8;
- бинарные массивы кодируются через Base64;
- версия формата — 3.

Проект ориентирован на:

- чтение и запись `.fc`-файлов;
- объектное представление сущностей Fidesys Case;
- строго типизированный Python API;
- устойчивый encode/decode-протокол;
- roundtrip-сериализацию без потери данных;
- удобство использования библиотеки внешними проектами.

Технические свойства проекта:

- Python ≥ 3.8;
- зависимость: `numpy ≥ 1.20`;
- публикуется на PyPI как `fc-model`;
- поддерживает PEP 561 через `py.typed`;
- публичный API реэкспортируется из корня пакета `fc_model`.

Основная ценность проекта — стабильное соответствие между:

- спецификацией формата `.fc`;
- объектной моделью Python;
- публичным API;
- encode/decode-сериализацией;
- тестами roundtrip и реальными `.fc`-файлами.

## 3. Назначение основных файлов и каталогов

### `src/fc_model/`

Основной код библиотеки.

Ключевые модули:

* `__init__.py` — корневой модуль, `FCModel`, реэкспорт публичного API, `__all__`;
* `fc_mesh.py` — `FCMesh`: узлы, элементы, Base64-массивы;
* `fc_elements.py` — `FCElement`, константы типов элементов;
* `fc_blocks.py` — `FCBlock`;
* `fc_materials.py` — `FCMaterial`, `FCMaterialProperty`, константы свойств;
* `fc_data.py` — `FCData`, `FCDependencyColumn`, константы dependency-типов;
* `fc_value.py` — `FCValue`: Base64 encode/decode массивов и формул;
* `fc_conditions.py` — `FCLoad`, `FCRestraint`, `FCInitialSet` и их константы;
* `fc_constraint.py` — `FCConstraint`, константы контактов, связей и периодических ГУ;
* `fc_coordinate_system.py` — `FCCoordinateSystem`, типы систем координат;
* `fc_property_tables.py` — `FCPropertyTable`, property table и beam section типы;
* `fc_receivers.py` — `FCReceiver`, типы receivers;
* `fc_set.py` — `FCSet`, nodesets и sidesets;
* `fc_addons.py` — вспомогательные утилиты, находящиеся в разработке.

### `tests/`

Тесты библиотеки.

Назначение ключевых тестов:

* `test_value.py` — `FCValue` encode/decode;
* `test_data.py` — `FCData` constant/formula/table;
* `test_materials.py` — `FCMaterial` и `FCMaterialProperty`;
* `test_mesh.py` — `FCMesh` и `FCElement`;
* `test_conditions.py` — `FCLoad`, `FCRestraint`, `FCInitialSet`;
* `test_model.py` — интеграция `FCModel`, helpers, encode/decode, file I/O;
* `test_constants.py` — публичные константы;
* `test_round_trip.py` — roundtrip-тесты с реальными `.fc`-файлами;
* `test_cube_sideset.py` — тесты кубов с sidesets;
* `data/` — тестовые `.fc`-файлы.

При изменении публичного API, encode/decode-логики, констант или структуры `.fc` агент обязан обновить или добавить тесты.

### `docs/`

Спецификации и сопровождающие документы.

Ключевой источник истины:

* `docs/FidesysCase.md` — полная изначальная спецификация формата `.fc`.

Документ может быть частично устаревшим и может не содержать некоторые ключи, уже поддержанные библиотекой. Если агент обнаруживает расхождение между библиотекой и спецификацией, он обязан явно сообщить об этом пользователю.

При расхождении библиотеки и спецификации:

* если спецификация описывает поведение, которого нет в библиотеке, это кандидат на реализацию;
* если библиотека содержит расширения, отсутствующие в спецификации, это допустимо только если они задокументированы в LLM-контексте, README, тестах или changelog;
* агент не должен молча считать код более авторитетным, чем спецификация.

### LLM-документы проекта

Если в проекте присутствуют следующие документы, агент должен использовать их как контекст:

* `AGENTS.md` — правила работы агента;
* `FC_MODEL_LLM_CONTEXT.md` — подробное описание API для LLM;
* `FC_MODEL_API_INDEX_TLDR.md` — компактный индекс API;
* `docs/FidesysCase.md` — спецификация формата `.fc`.

Если эти документы расходятся, агент должен сообщить о расхождении и предложить синхронизацию.

### `work/`

Рабочие документы взаимодействия пользователя и LLM.

`work/` используется для:

* планов разработки;
* анализа вариантов;
* промежуточных проектных решений;
* вопросов к пользователю;
* документов, созданных в одной LLM-сессии и исполняемых в другой;
* подготовки сложных изменений до редактирования кода.

Если задача нетривиальна, затрагивает публичный API, формат `.fc`, encode/decode-протокол, совместимость или несколько подсистем, агент по умолчанию сначала создаёт план в `work/`, а не сразу меняет код.

Формат имени:

```text
<version>_<date>_<title>.md
```

Где:

* `<version>` строится из первых двух чисел файла `VERSION`;
* если `VERSION == 0.8.12`, то `<version> == 0_8`;
* `<date>` записывается в формате `YYYY-MM-DD`;
* `<title>` — короткое имя латиницей или транслитерацией.

Пример:

```text
0_8_2026-05-10_decode_constraints_plan.md
```

Запрещено создавать промежуточные документы в корне проекта.

## 5. Архитектурные инварианты

Агент обязан сохранять следующие инварианты:

* основной код находится в `src/fc_model/`;
* публичный API импортируется из корня пакета `fc_model`;
* внутренние модули `fc_model.fc_*` не должны использоваться пользователями библиотеки напрямую;
* полный публичный API должен быть отражён в `__all__` в `src/fc_model/__init__.py`;
* библиотека остаётся PEP 561-совместимой;
* доменные классы используют единый encode/decode-протокол;
* JSON-ready-структуры должны оставаться совместимыми с форматом `.fc`;
* Base64-массивы обрабатываются через `FCValue`, а не через локальные ручные хаки;
* roundtrip-сериализация должна сохранять данные без потери смысла;
* тестовые `.fc`-файлы в `tests/data/` не должны портиться ручным редактированием;
* расширения формата, отсутствующие в `docs/FidesysCase.md`, должны быть покрыты тестами и документацией;
* изменения публичных констант должны сопровождаться обновлением тестов и LLM-документов.

## 6. Правила импортов

Импорт публичных сущностей библиотеки должен выполняться только из корня пакета.

Правильно:

```python
from fc_model import FCModel, FCMaterial, FCData
```

Неправильно:

```python
from fc_model.fc_materials import FCMaterial
from fc_model.fc_data import FCData
```

Исключение допустимо только внутри самой библиотеки `src/fc_model/`, где модули могут импортировать друг друга согласно архитектуре проекта.

Тесты, примеры и пользовательский код должны использовать корневой импорт.

## 7. Encode/decode-протокол

Каждый доменный класс должен реализовывать пару методов:

```python
@classmethod
def decode(cls, src_data: FCSrc*) -> Entity: ...

def encode(self) -> FCSrc*: ...
```

Смысл протокола:

* `decode` строит объектную модель из TypedDict/JSON-ready данных;
* `encode` возвращает JSON-ready структуру для записи `.fc`;
* encode/decode должны быть согласованы друг с другом;
* изменение decode-логики почти всегда требует проверки encode-логики и roundtrip-тестов.

Корневой класс `FCModel` поддерживает:

```python
FCModel.load(path) -> FCModel
FCModel.decode(data) -> FCModel
model.encode() -> dict
model.save(path) -> None
```

Запрещено использовать `FCModel("path")` как способ загрузки файла. Правильно:

```python
model = FCModel.load("case.fc")
```

## 8. Антипаттерны

Запрещено:

* размещать основной код пакета вне `src/`;
* создавать временные скрипты или промежуточные планы в корне проекта;
* импортировать публичные сущности из внутренних модулей в тестах, примерах и пользовательском коде;
* хардкодить числовые коды вместо использования публичных констант;
* обходить `encode()` / `decode()` при сериализации;
* обходить `FCValue` при работе с Base64-массивами;
* напрямую редактировать бинарные Base64-поля;
* использовать `FCModel("path")` вместо `FCModel.load("path")`;
* добавлять `Any` как способ скрыть ошибку типизации;
* менять публичный API без обновления `__all__`, тестов и LLM-документов;
* молча игнорировать расхождение между спецификацией и кодом;
* менять `VERSION` без прямого разрешения пользователя;
* выполнять git-команды изменения без прямого разрешения пользователя.

# II. История проекта

## 9. VERSION

Файл `VERSION` содержит текущую версию проекта.

`VERSION` определяет:

* нумерацию новых блоков `CHANGELOG.md`;
* префиксы документов в `work/`;
* текущую линию разработки.

Агент не меняет и не инкрементирует `VERSION`.

Исключение возможно только при прямом, недвусмысленном указании пользователя.

Если агенту нужна версия для имени файла в `work/`, он читает `VERSION` и берёт первые два числа.

Пример:

```text
VERSION = 0.8.12
work prefix = 0_8
```

## 10. CHANGELOG.md

Все изменения агента должны быть отражены в `CHANGELOG.md`.

`CHANGELOG.md` — это буфер передачи ответственности от агента к пользователю и далее к `git+`.

Агент отвечает за содержательное описание изменения в блоке `[Unreleased]`.
Пользователь отвечает за ревью diff и закрытие изменения в git-историю.

### 10.1. Проверка `[Unreleased]`

Перед завершением задачи агент обязан проверить, есть ли в `CHANGELOG.md` блок с меткой `[Unreleased]`.

Если блок существует, агент добавляет запись в подходящую секцию этого блока.

Если блока нет, агент создаёт новый блок над последним существующим блоком.

### 10.2. Создание нового блока

При создании нового блока агент:

1. читает текущую версию из `VERSION`;
2. инкрементирует patch-часть только в заголовке changelog;
3. не меняет файл `VERSION`;
4. создаёт заголовок:

```markdown
## X.Y.Z [YYYY-MM-DD HH:MM:SS] [Unreleased]
```

Пример:

```markdown
## 0.8.13 [2026-05-10 18:30:00] [Unreleased]
```

### 10.3. Секции changelog

Рекомендуемые секции:

```markdown
### Added

### Changed

### Fixed

### Removed

### Docs

### Tests

### Internal
```

Допустимы русские секции, если текущий `CHANGELOG.md` уже ведётся на русском. Агент должен сохранять существующий стиль файла.

### 10.4. Требование к записи

Запись в `CHANGELOG.md` должна быть достаточно содержательной, чтобы пользователь понял смысл изменения без чтения полного diff.

При этом changelog не заменяет ревью diff.

Хорошая запись отвечает на вопросы:

* что изменено;
* зачем изменено;
* какие публичные или архитектурные контракты затронуты;
* какие тесты, docs или сценарии обновлены;
* какие проверки выполнены или должны быть выполнены пользователем.

## 11. CHANGELOG_ARCHIVE.md и релизная свёртка

При переходе проекта на новую крупную линию версий по прямому указанию пользователя агент:

1. читает текущий `CHANGELOG.md`;
2. выделяет важные изменения завершённой линии;
3. переносит подробное содержимое старого changelog в `CHANGELOG_ARCHIVE.md`, не дублируя уже перенесённые блоки;
4. заменяет рабочий `CHANGELOG.md` коротким релизным описанием новой версии;
5. не меняет `VERSION`, если пользователь не дал прямое разрешение;
6. фиксирует в релизном описании изменения правил, структуры, документации и архитектуры, если они были;
7. после релизной свёртки снова ведёт работу через новый `[Unreleased]` блок.

## 12. work-документы как основа диалога

Для сложных задач агент по умолчанию начинает с плана в `work/`.

Сложной считается задача, которая:

* создаёт новый публичный API или меняет семантику существующего API;
* меняет формат `.fc`;
* меняет encode/decode-протокол;
* меняет обработку Base64-массивов;
* меняет модель материалов, данных, условий, mesh, constraints, receivers или property tables;
* требует проектного выбора из нескольких альтернатив;
* требует серии тестов или обновления LLM-документов;
* может повлиять на обратную совместимость.

Не является сложной задача, которая затрагивает много файлов, но не требует проектного решения, например механическое переименование после согласованного решения.

### 12.1. План разработки

План в `work/` должен содержать:

```markdown
# <Название плана>

## Контекст

Что известно из задачи пользователя и текущего состояния проекта.

## Цель

Какой результат должен быть получен.

## Источники истины

Какие документы и файлы нужно учитывать.

## Предполагаемые изменения

Какие файлы или подсистемы могут быть затронуты.

## Вопросы к пользователю

Все неясные аспекты задачи.

## План работ

Пошаговая схема реализации.

## Проверки

Какие проверки выполнит агент, а какие должен выполнить пользователь.

## Риски

Что может сломаться или потребовать отдельного решения.
```

Во время планирования агент обязан задавать вопросы по неясным аспектам внутри документа плана.

Если пользователь одобрил или скорректировал план, агент исполняет уже согласованный план.

Если задача простая и локальная, агент может не создавать `work/`-план, но всё равно обязан обновить `CHANGELOG.md` после изменения.

## 13. Git-дисциплина

```bash

git log
git show
git blame
git rev-parse HEAD
git branch --show-current
```

Запрещены команды, изменяющие рабочее дерево, индекс, ветки, теги, историю или удалённые репозитории:

```bash
git add
git commit
git restore
git checkout
git switch
git reset
git revert
git merge
git rebase
git cherry-pick
git tag
git stash
git push

```

Исключение возможно только по прямому, недвусмысленному приказу пользователя.

Даже если пользователь использует `git+`, агент не запускает `git+` самостоятельно, если пользователь явно не приказал это сделать.

## 14. GitPlus-контракт

FCModel использует GitPlus-идеологию:

1. Агент готовит изменение.
2. Агент описывает изменение в `CHANGELOG.md` под `[Unreleased]`.
3. Пользователь смотрит diff в VSCode или другом инструменте.
4. Пользователь принимает, корректирует или отклоняет изменение.
5. Пользователь запускает команду `git+`.
6. `git+` обновляет `VERSION`, снимает `[Unreleased]` и создаёт унифицированный commit.

Агент не должен нарушать этот контракт.

# III. Правила разработки

## 15. Общий рабочий цикл агента

Перед изменением проекта агент должен:

1. прочитать `AGENTS.md`;
2. определить, какие источники истины относятся к задаче;
3. прочитать релевантные документы из `docs/` и LLM-контекста;
4. проверить текущее состояние файлов;
5. при необходимости посмотреть `git status` и `git diff`;
6. для сложных задач подготовить план в `work/`;
7. получить пользовательское одобрение плана, если задача требует проектного решения.

Во время изменения агент должен:

1. вносить минимальное связное изменение;
2. не смешивать независимые задачи без необходимости;
3. поддерживать синхронность кода, спецификации, README, LLM-документов и тестов;
4. не разрушать публичный API;
5. фиксировать обнаруженные расхождения между кодом и docs.

После изменения агент должен:

1. выполнить разрешённые проверки;
2. обновить `CHANGELOG.md`;
3. сообщить пользователю итог;
4. остановиться, не выполняя commit.

## 16. Минимальное связное изменение

Агент должен стремиться к минимальному связному изменению:

```text
одна цель → один связный набор правок → одна запись в CHANGELOG.md
```

Если задача распадается на несколько независимых изменений, агент должен:

* предложить разбиение;
* либо явно сгруппировать изменения в плане и changelog;
* не делать “заодно” архитектурно не связанные исправления без необходимости.

## 17. Типизация

Проект строго типизирован.

После изменения кода агент должен проверить типизацию, если это технически возможно:

```bash
python -m pyright src/fc_model tests
```

Если конфигурация проекта задаёт другую команду pyright, агент должен использовать команду из `pyproject.toml` или существующей документации проекта.

Все ошибки типов должны быть исправлены перед завершением задачи.

Исключения:

* ложные срабатывания от неполных stubs сторонних библиотек;
* ограничения сторонних библиотек, где строгая типизация невозможна без чрезмерного шума.

Исключения должны быть явно документированы в `pyproject.toml` или в финальном отчёте агента.

`Any` не должен использоваться как способ скрыть ошибку типизации. Предпочтительны:

* конкретные типы;
* `TypedDict`;
* `Protocol`;
* `Union` / `|`;
* generic-типы;
* локальные narrow-проверки.

## 18. Pytest

После изменения кода или тестов агент должен проверить работоспособность unit-тестов:

```bash
python -m pytest tests/ -v
```

Если тесты не удаётся запустить из-за отсутствующих зависимостей, окружения или тестовых данных, агент обязан указать это в финальном отчёте и дать команду для пользовательского запуска.

При изменении API агент должен либо обновить тесты под новый согласованный контракт, либо исправить код, если тесты выявляют регрессию.

## 19. Документация и синхронизация

Если изменение затрагивает публичный API, формат `.fc`, encode/decode-протокол, константы, README, LLM-контекст или спецификацию, агент обязан обновить соответствующую документацию.

Правило:

```text
код изменил смысл → docs и LLM-контекст должны быть проверены
код изменил публичный контракт → docs, README, __all__ и тесты должны быть обновлены
код выявил расхождение с docs → пользователь должен быть уведомлён
```

Если агент не уверен, что документацию нужно менять, он должен явно указать это в финальном отчёте.

## 20. Финальный отчёт агента

После завершения задачи агент обязан сообщить пользователю итог.

Для тривиальных задач достаточно краткого итога и обновления `CHANGELOG.md`.

Для содержательных задач отчёт должен включать:

1. какие файлы изменены и что именно изменено;
2. какие источники истины были проверены;
3. обнаруженные расхождения между кодом и docs;
4. выполненные проверки и проверки, которые должен выполнить пользователь;
5. что добавлено в `CHANGELOG.md`;
6. риски или нерешённые вопросы.

Агент не должен предлагать выполнить commit самостоятельно.

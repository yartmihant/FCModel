# AGENTS.md — Руководство для ИИ-агентов

Этот документ содержит правила и контекст для ИИ-ассистентов, работающих с проектом **fc_model**.

## О проекте

**fc_model** — строго типизированная Python-библиотека объектной модели формата Fidesys Case (`.fc`).  
Формат: JSON с Base64-кодированными бинарными массивами, кодировка UTF-8, версия формата — 3.

- Python ≥ 3.8, зависимость: `numpy ≥ 1.20`.
- Публикуется на PyPI: [fc-model](https://pypi.org/project/fc-model/).
- PEP 561 (`py.typed`) — строгая типизация.

## Правила импорта

Импорт **только** из корня пакета:

```python
from fc_model import FCModel, FCMaterial, FCData  # ✓
from fc_model.fc_materials import FCMaterial       # ✗ запрещено
```

Полный список публичных символов — в `__all__` в `src/fc_model/__init__.py`.

## Паттерн encode/decode

Каждый доменный класс реализует:
- `@classmethod decode(cls, src_data: FCSrc*) -> Entity` — из TypedDict в объект
- `encode(self) -> FCSrc*` — из объекта в TypedDict (JSON-ready)

Корневой FCModel:
- `FCModel.load(path) -> FCModel` — чтение .fc файла
- `FCModel.decode(data) -> FCModel` — из dict
- `model.encode() -> dict` — сериализация
- `model.save(path)` — запись .fc файла

## Список изменений проекта (`CHANGELOG.md`)

Если вы — ИИ-ассистент, работающий над этим проектом, вы обязаны соблюдать следующие правила управления изменениями и версиями.

### 1. Фиксация изменений (Changelog)

Вы должны фиксировать проделанную работу в файле `CHANGELOG.md`.

- **Проверка наличия блока [Unreleased]:**
  Проверьте, существует ли в `CHANGELOG.md` заголовок версии с меткой `[Unreleased]`.

- **Если блок [Unreleased] существует:**
  Добавьте описание ваших изменений в соответствующие секции (Added, Changed, Fixed и т.д.) этого блока.

- **Если блока [Unreleased] нет:**
  Создайте новый блок версии над последним существующим блоком.

### 2. Создание нового блока версии

При создании нового блока:
1. Считайте текущую версию из файла `VERSION` (например, `1.2.3`).
2. Инкрементируйте патч-версию (последнее число) на +1 (станет `1.2.4`), не меняя `VERSION`.
3. Добавьте в `CHANGELOG.md` новый заголовок вида:
   ```markdown
   ## 1.2.4 [YYYY-MM-DD HH:MM:SS] [Unreleased]
   ```
   (подставьте актуальную дату и время).
4. Ниже добавьте категории изменений (Добавлено, Изменено, Исправлено, Убрано) и описание проделанной вами и пользователем незафиксированной работы.

### 3. Релиз мажорной версии

При переходе проекта на новую крупную линию версий (например, `0.7` → `0.8`) по прямому указанию пользователя:

1. Прочитайте текущий `CHANGELOG.md` и выделите самые важные изменения завершённой линии версии.
2. Перенесите подробное содержимое старого `CHANGELOG.md` в `CHANGELOG_ARCHIVE.md`, сохранив уже существующий архив и не дублируя ранее перенесённые блоки.
3. Замените рабочий `CHANGELOG.md` коротким релизным описанием новой версии: заголовок текущей версии из `VERSION`, дата/время релиза, краткая фиксация самого факта релиза и сводка ключевых изменений.
4. Не меняйте `VERSION` самостоятельно: пользователь обновляет этот файл перед релизной свёрткой или отдельно разрешает его изменение.
5. Если в ходе релиза обновлялись правила работы, структура проекта или документы, зафиксируйте это в релизном описании.
6. После релизной свёртки дальнейшие изменения снова ведутся по обычным правилам через новый `[Unreleased]` блок.

## Запрещено использовать git-команды изменения

**Важно**: ИИ-агентам запрещено выполнять git-команды, изменяющие состояние репозитория, рабочее дерево, индекс, ветки, теги, историю коммитов или удалённые репозитории. Это инструмент контроля пользователя над проектом, и право его использования остается исключительно за пользователем.

Запрещены, в частности:
`git add`, `git commit`, `git restore`, `git checkout`, `git switch`, `git reset`, `git revert`, `git merge`, `git rebase`, `git cherry-pick`, `git tag`, `git stash`, `git push`, `git pull`, `git fetch`. Применение таких команд допустимо только при явном прямом недвусмысленном приказе пользователя.

Разрешены read-only команды для анализа текущего состояния проекта, например:
`git status`, `git diff`, `git diff --staged`, `git log`, `git show`, `git blame`, `git rev-parse HEAD`, `git branch --show-current`. `git diff` и `git status` предпочтительны, когда агенту нужно понять контекст уже внесённых локальных изменений.

Даже в разрешённом режиме git используется только как инструмент чтения и анализа.
Любые действия по фиксации, откату, синхронизации или переписыванию истории остаются исключительно за пользователем.

## Качество кода

### Проверка типов (pyright)

После написания или изменения кода следует проверять его через pyright.

Все ошибки типов должны быть исправлены перед завершением задачи. Исключения:

- Ложные срабатывания от неполных stubs сторонних библиотек (numba, matplotlib) — они должны быть задокументированы в секции `[tool.pyright]` файла `pyproject.toml` с комментарием-пояснением.
- Правило `Any` не должно использоваться как способ обойти ошибку типизации — вместо этого следует применять конкретные типы, `TypedDict`, `Union` или корректные аннотации.

### Проверка тестов (pytest)

После написания или изменения кода следует проверять работоспособность unit-тестов. При наличии проблем нужно либо исправить тест (например, при изменении API), либо исправить код.

```bash
python -m pytest tests/ -v
```

## Спецификация формата

Источник истины — `docs/FidesysCase.md`. При расхождении библиотеки и спецификации:
- Если спецификация описывает поведение, которого нет в библиотеке — это кандидат на реализацию.
- Если библиотека содержит расширения, отсутствующие в спецификации (BAR2/3, CABLE2/3, VOIGT_*, VP, VS, swelling) — это допустимые расширения, задокументированные в LLM-контексте.

## Антипаттерны (запрещено)

- Хардкодить числовые коды вместо использования констант (`FC_LOADS_TYPES_KEYS`, и т.д.)
- Импортировать из внутренних модулей (`fc_model.fc_*`)
- Обходить `encode()`/`save()` для сериализации
- Напрямую редактировать бинарные Base64-поля без `FCValue`
- Использовать `FCModel("path")` — правильно: `FCModel.load("path")`
